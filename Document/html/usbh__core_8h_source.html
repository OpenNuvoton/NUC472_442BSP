712 /*
713  * Calling this entity a "pipe" is glorifying it. A USB pipe
714  * is something embarrassingly simple: it basically consists
715  * of the following information:
716  * - device number (7 bits)
717  * - endpoint number (4 bits)
718  * - current Data0/1 state (1 bit)
719  * - direction (1 bit)
720  * - speed (1 bit)
721  * - max packet size (2 bits: 8, 16, 32 or 64) [Historical; now gone.]
722  * - pipe type (2 bits: control, interrupt, bulk, isochronous)
723  *
724  * That's 18 bits. Really. Nothing more. And the USB people have
725  * documented these eighteen bits as some kind of glorious
726  * virtual data structure.
727  *
728  * Let's not fall in that trap. We'll just encode it as a simple
729  * uint32_t. The encoding is:
730  *
731  * - max size: bits 0-1 (00 = 8, 01 = 16, 10 = 32, 11 = 64) [Historical; now gone.]
732  * - direction: bit 7 (0 = Host-to-Device [Out], 1 = Device-to-Host [In])
733  * - device: bits 8-14
734  * - endpoint: bits 15-18
735  * - Data0/1: bit 19
736  * - speed: bit 26 (0 = Full, 1 = Low Speed)
737  * - pipe type: bits 30-31 (00 = isochronous, 01 = interrupt, 10 = control, 11 = bulk)
738  *
739  * Why? Because it's arbitrary, and whatever encoding we select is really
740  * up to us. This one happens to share a lot of bit positions with the UHCI
741  * specification, so that much of the uhci driver can just mask the bits
742  * appropriately.
743  */
